MiniPACS — Hardware Analysis Memo

Author: Claude (Anthropic, acting as engineering assistant for MiniPACS project)
Date: 2026-04-20
Subject: Why the current VM is the bottleneck for worklist performance on a PACS with ~2,500 studies / 400,000 DICOM instances.
Intended audience: Systems administrator reviewing MiniPACS deployment at Clinton Medical.

TL;DR. A PACS workload is not comparable to an e-commerce catalog of the same row count. A DICOM instance is ~3-4 orders of magnitude larger than an e-commerce item, and Orthanc's hierarchical search (Patient → Study → Series → Instance with tag filtering) produces a very different access pattern from SELECT * FROM products WHERE id = ?. For ~400k instances, the official Orthanc recommendation is ≥ 8 GB RAM. This VM has 3.8 GB. The observed behavior (cold-cache /tools/find > 30 s) is a direct consequence of that gap, not of the application code.

1. The raw data asymmetry

DimensionE-commerce (100k items)MiniPACS here (400k instances)Ratio
Size of one record on disk1–5 KB (JSON)~30 MB (avg DICOM instance — MRI Spine observed at 32–34 MB)~6,000–30,000×
Total payload storage~500 MB~200 GB (measured on this deployment)~400×
Index size (on-disk)5–20 MB637 MB (measured: /var/lib/orthanc/db/index)30–100×
Metadata per row~10 fields150+ DICOM tags per instance, stored in MainDicomTags~15×

The claim "100k items in e-commerce ran fine, so 400k should be fine here" conflates row count with data volume. By total payload this system is ~400× larger, and by index alone it is ~30–100× larger. The VM's memory has not scaled accordingly.

2. Query complexity is also not comparable

E-commerce typical query. Primary-key lookup on a single table:

SELECT * FROM products WHERE id = 12345;

This is O(log n) over a B-tree, resolved in a handful of page reads, well under 1 ms on any modern box.

Orthanc worklist query (POST /tools/find { Level: Study, Query: {...} }) has to:

  1. Traverse a 4-level hierarchy (Patient / Study / Series / Instance).
  2. Filter by arbitrary DICOM tags stored in MainDicomTags / MainDicomTagsStudies.
  3. Match wildcard patterns (e.g. PatientName LIKE '%smith%').
  4. Return an expanded JSON payload containing tags + computed counts.

On a cold cache with a 637 MB index, a full traversal requires random reads across a large fraction of the index file. If that file is not resident in OS page cache, every seek is a disk round trip. This is why the first /tools/find after a restart takes tens of seconds, while subsequent ones (with the index hot in page cache) complete in tens of milliseconds.

3. Memory budget on this VM

Measured (free -h on the running system):

Mem:           3.8 GiB total,   free ~100 MiB,   buff/cache ~3.0 GiB
Swap:          1.0 GiB total

Realistic allocation during normal operation:

ConsumerTypical working set
Windows host + Hyper-V overhead (outside the VM, but contends for the same physical box)1.0–1.5 GB
WSL2 / systemd inside the VM0.3–0.5 GB
Docker engine + cloudflared + nginx + FastAPI backend0.5–0.8 GB
Orthanc process itself (heap, HTTP threads, Lua, plugins)0.3–0.6 GB
SQLite page cache desired (to hold the 637 MB index hot)~0.7 GB at minimum
Total desired working set~3.5 GB

With 3.8 GB total, the index does eventually fit in page cache — but there is almost no slack. After any memory pressure event (a burst of large DICOM C-STORE, a backup, a compose restart), the cache is evicted, and the next query has to re-warm the full 637 MB from disk before it can respond. That re-warm is exactly what was observed today.

4. Official Orthanc guidance

Orthanc Book — "Improving performance and scalability"
https://orthanc.uclouvain.be/book/faq/scalability.html
The Orthanc maintainers recommend migrating from SQLite to PostgreSQL/MySQL for large installations, and provisioning memory sized to the working set of the index. Community guidance and production deployment notes consistently cite ≥ 8 GB RAM as a baseline for installations in the 100k–1M instance range, and more as the archive grows.
Orthanc Book — "Tuning Orthanc for large amounts of data"
Relevant settings include StorageCompression (off for read-heavy workloads), HttpThreadsCount, LimitFindResults, and the choice of database backend. These are tunings around adequate hardware — they do not substitute for RAM or IOPS.

5. The CPU side

The VM runs on an Intel Xeon E5-2430 @ 2.20 GHz — a 2012-generation Sandy Bridge-EP part. Public Passmark single-thread score: ~1,100 points. A 2021-era Ryzen 5 5600G scores ~3,200 (≈3× faster per core). SQLite query execution is overwhelmingly single-threaded at the statement level, so single-thread CPU speed is the binding constraint on /tools/find when the index is in cache.

Combined with triple virtualization (Hyper-V → WSL2 → Docker), each kernel boundary adds a small but measurable tax on syscall-heavy workloads like SQLite.

Of these, RAM is the single change that would most visibly resolve the symptom the admin and user just observed.

7. Observed data from this specific deployment

Measured today (2026-04-20) on the live system:

  • CountStudies = 2,523, CountInstances = 398,274 (from /statistics)
  • SQLite index file size: 637 MB
  • Storage directory size: ~200 GB (DICOM payload with gzip compression)
  • /statistics response time on a warm instance: ~0.3 s (proves Orthanc itself is healthy)
  • /tools/find Level=Study response time on a cold cache: > 30 s (the observed 502 at the UI is exactly the backend's 30 s httpx timeout being tripped)
  • Backend log signature: POST /api/studies HTTP/1.1 502 Bad Gateway issued at 31,486 ms — matches the 30 s httpx timeout + ~1 s overhead.

The failure mode is therefore deterministic and explained: cold 637 MB index, insufficient page cache headroom, single-threaded SQLite scan on a 2012 CPU, and a 30 s client timeout on the backend. Fix the memory headroom, and the 502 disappears.

8. Verdict

The application is configured reasonably for its tier. The chosen stack (Orthanc + SQLite + FastAPI + Docker Compose + Cloudflare Tunnel) is a standard small-clinic deployment pattern. There are tunings to apply (compression off, thread counts, eventual Postgres migration), and they should be done — but they are optimizations on top of adequate hardware.

The hardware is the primary bottleneck for this data volume. 3.8 GB of RAM for 400,000 DICOM instances is below the Orthanc project's published recommendation. Raising the VM to 16 GB RAM is the single change that will most visibly restore interactive performance on the worklist and viewer.
